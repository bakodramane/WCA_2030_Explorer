## Role & Objective

You are building a **production-quality, offline-first Progressive Web App** (PWA) that answers questions exclusively from the *World Programme for the Census of Agriculture 2030* (WCA 2030) guidelines. Every answer must be verbatim extracted text from the guidelines, accompanied by a precise section title and page number. If the guidelines do not support an answer, the app must say so explicitly and list what was searched.

Work through the phases below in order. **Complete and confirm each phase before starting the next.** At the end of every phase, print a summary of files created or modified.

## Hard Constraints

1. **Answers are extracted text only** — never paraphrased or generated. The retrieved chunk is the answer.
2. **No external API calls at runtime** — the app must work with zero internet access after first load.
3. **Guardrail is mandatory** — below the confidence threshold, the app returns: *"This question could not be answered from the WCA 2030 guidelines. Sections searched: [list]."*
4. **Every answer cites** the source chunk's section title and page number.
5. **No generative model at runtime** — do not integrate any LLM inference at query time.

## Phase 2 — PDF Extraction & Chunking (`scripts/chunk.ts`)

### Step 2a — PDF text extraction

Use `pdf-parse` to extract text with page boundaries preserved:

```ts
import pdf from 'pdf-parse';
import fs from 'node:fs';
import path from 'node:path';

const pdfPath = path.join(process.cwd(), 'source', 'Census-2030_EN-DTP-9.pdf');
const dataBuffer = fs.readFileSync(pdfPath);

// Capture per-page text using the pagerender callback
const pages: { pageNum: number; text: string }[] = [];

await pdf(dataBuffer, {
  pagerender: (pageData) => {
    return pageData.getTextContent().then((content: any) => {
      const text = content.items.map((item: any) => item.str).join(' ');
      pages.push({ pageNum: pageData.pageNumber, text });
      return text;
    });
  },
});
```

If `pagerender` is unsupported by the installed version of `pdf-parse`, fall back to extracting `data.text` and splitting on form-feed characters (`\f`) to approximate page boundaries, numbering each segment sequentially.

### Step 2b — Chunking

After extraction, chunk the page array using this strategy:

- Detect section headings using regex patterns that match: numbered headings (`4.1`, `7.2.3`), `CHAPTER`, `ANNEX`, `GLOSSARY`, and all-caps lines of 3–10 words.
- Split each section into overlapping paragraphs of **200–350 words**. Overlap consecutive chunks by **50 words** (slide the window).
- Assign each chunk:
  ```ts
  interface Chunk {
    id: string;          // e.g. "ch4-s2-p3"
    sectionTitle: string;
    pageRef: number;     // page number from pdf-parse
    text: string;        // verbatim extracted text
    priority: 'high' | 'normal';
  }
  ```
- Set `priority: 'high'` for chunks whose `pageRef` falls in the four key regions (verify page ranges against your PDF copy and update the landmark table at the top of this file).

### Step 2c — Output & validation

- Write `./src/data/chunks-raw.json`.
- Log: total chunks, high-priority count, average chunk word count, page range covered.
- Write unit tests in `tests/chunking.test.ts` asserting:

  - Every chunk has a non-empty `sectionTitle` and a positive `pageRef`.

## Phase 4 — Retrieval Engine (`src/engine/retrieval.ts`)

Implement a `RetrievalEngine` class:

### Types (`src/engine/types.ts`)
```ts
export interface Chunk {
  id: string;
  sectionTitle: string;
  pageRef: number;
  text: string;
  priority: 'high' | 'normal';
  embedding: number[];
}

export interface RankedResult {
  chunk: Chunk;
  score: number;
  matchType: 'semantic' | 'lexical';
}
```

### `async init(): Promise<void>`
- Fetch and parse `./data/chunks.json`. Convert each `embedding` array to `Float32Array` for performance.
- Load `Xenova/all-MiniLM-L6-v2` from the cached `./models/` path (set `env.localModelPath` and `env.allowRemoteModels = false` to enforce offline use).
- Build a `minisearch` index over all chunk `text` fields (fields: `text`, `sectionTitle`).

### `async semanticSearch(query: string, topK = 5): Promise<RankedResult[]>`
- Encode the query with `normalize: true`.
- Compute cosine similarity using a `Float32Array` dot product loop (vectors are already normalised, so dot product equals cosine similarity).
- Multiply scores of `priority: 'high'` chunks by `1.15` before ranking.
- Return the top-K results sorted by score descending.

### `lexicalSearch(query: string, topK = 5): RankedResult[]`
- Return top-K `minisearch` results as `RankedResult[]` with `matchType: 'lexical'`.

Write unit tests in `tests/retrieval.test.ts` with mock embeddings verifying: correct ranking order, score boosting for high-priority chunks, and lexical fallback returns results.

**Phase 4 complete when:** tests pass and the engine initialises without errors in a browser console.

## Phase 5 — Guardrail (`src/engine/guardrail.ts`)

```ts
export const CONFIDENCE_THRESHOLD = 0.42; // tune after manual testing

export interface GuardrailResponse {
  answered: boolean;
  results?: RankedResult[];
  message?: string;
  sectionsSearched?: string[];
}

export function evaluate(results: RankedResult[], query: string): GuardrailResponse {
  const passing = results.filter(r => r.score >= CONFIDENCE_THRESHOLD);
  if (passing.length === 0) {
    return {
      answered: false,
      message: 'This question could not be answered from the WCA 2030 guidelines.',
      sectionsSearched: results.map(r => r.chunk.sectionTitle),
    };
  }
  return { answered: true, results: passing };
}
```

- Expose `CONFIDENCE_THRESHOLD` as a developer-tunable value: read from `localStorage.getItem('wca_threshold')` if present, falling back to `0.42`.
- Write `tests/guardrail.test.ts` covering: best score above threshold → answered; best score below → not answered; empty results → not answered with a non-empty `sectionsSearched` list.

**Phase 5 complete when:** all three guardrail tests pass.
